# umjik — 제1원칙 분석

이 문서는 umjik의 모든 결정이 어떤 근본 전제 위에 서 있는지를 기록합니다.
구현 중 방향이 흔들릴 때 이 문서로 돌아와 검증합니다.

---

## 출발점

모든 건 하나의 관찰에서 시작합니다.

**"React Native에서 애니메이션을 만드는 건 필요 이상으로 어렵다."**

이걸 분해하면:

- **사실 1**: 모바일 앱에는 애니메이션이 필요하다. 사용자는 상태 변화를 시각적 전환으로 이해한다.
- **사실 2**: RN은 성능 문제(JS 스레드 / UI 스레드 분리)로 인해 애니메이션 구현이 복잡해졌다. Reanimated는 worklet으로 성능을 해결했지만, 개발자에게 높은 인지적 비용을 요구한다.
- **사실 3**: 복잡성의 원인은 "성능"과 "표현력" 사이의 트레이드오프다. 쉽게 쓰려면 성능을 포기해야 하고(내장 Animated), 성능을 원하면 복잡해야 한다(Reanimated).

핵심 질문: **이 트레이드오프는 필연적인가, 아니면 추상화 레이어가 부족해서 생긴 문제인가?**

framer-motion이 답을 준다. 웹에서도 고성능 애니메이션은 저수준 API를 다뤄야 했지만, framer-motion이 선언적 추상화를 올려서 트레이드오프를 해소했다. **문제는 엔진이 아니라 인터페이스에 있다.** Reanimated라는 엔진은 이미 있고, 부족한 건 그 위의 선언적 인터페이스다.

---

## 핵심 전제들

### 전제 1: 문제는 엔진이 아니라 인터페이스에 있다

Reanimated는 충분히 좋은 엔진이다. 부족한 건 선언적 추상화다.

- **검증 상태**: 안전. Reanimated는 Shopify, Expo 등이 후원하는 검증된 엔진.
- **리스크**: Reanimated 자체에 근본적 성능 문제가 있다면 umjik도 같은 문제를 가진다. 하지만 현재 그런 증거는 없다.

### 전제 2: framer-motion의 API는 선언적 애니메이션의 좋은 표준이다

자체 API를 만드는 대신 framer-motion API를 따르는 이유:

- umjik의 핵심 문제는 "API 디자인"이 아니라 "RN에서 선언적 애니메이션이 없다"는 것. API 디자인에 에너지를 쓰면 핵심 문제에서 분산된다. Moti가 이 경로를 갔고, 좋은 API를 만들었지만 유지보수가 멈췄다.
- API 디자인에는 네트워크 효과가 있다. 월 3,000만 다운로드 생태계에 편승하는 게 0에서 시작하는 것보다 합리적이다.
- LLM 시대에 "학습 데이터 풍부함"은 실질적 가치.

**검증 상태**: 합리적이나 주의 필요.
**리스크**: framer-motion API가 웹 DOM에 지나치게 결합된 부분이 있다면 RN에서 어색해질 수 있다. 구현하면서 지속 검증해야 한다.

### 전제 3: 타겟 사용자는 이미 Reanimated를 쓰고 있다

Reanimated를 peer dependency로 두는 이유:

- framer-motion 호환을 위해 모든 속성의 애니메이션을 지원해야 하고, 내장 Animated로는 이 제약을 만족시킬 수 없다.
- 성능을 중시하는 RN 프로젝트는 이미 Reanimated를 사용하고 있을 확률이 높다 (npm 주간 ~150만 다운로드).

**검증 상태**: 합리적.
**리스크**: Reanimated를 안 쓰는 프로젝트에서 umjik만을 위해 설치하는 건 부담. 장기적으로 fallback 경로를 고려할 수 있다.

### 전제 4: 호환 API의 네트워크 효과가 자체 API의 유연성보다 가치 있다

**검증 상태**: 현재 안전. framer-motion은 월 3,000만 다운로드로 성장 중.
**리스크**: framer-motion이 쇠퇴하거나 API를 급격히 바꾸면 흔들린다.

### 전제 5: 추상화 레이어의 비용이 DX 이점보다 작다

Engine의 변환 로직이 매 렌더마다 실행되면 오버헤드가 될 수 있다.

**검증 상태**: 미검증. 구현 후 벤치마크 필요.
**리스크**: 메모이제이션으로 해결 가능하지만, 측정 전까지는 가정.

## 아키텍처 결정의 근거

### 3 레이어 분리

분리의 근본 이유: **변경의 이유가 다른 것들을 분리한다.**

- Public API가 바뀌는 이유: framer-motion이 API를 바꿀 때
- Engine이 바뀌는 이유: 변환 로직 버그나 새 속성 지원
- Adapter가 바뀌는 이유: Reanimated 메이저 업데이트

이 세 가지는 독립적인 변경 이유이므로 분리가 맞다.

**주의**: MVP 단계에서 세 레이어의 경계가 실제로 얼마나 뚜렷할지는 구현하면서 검증해야 한다. Adapter가 React hook이어야 한다는 제약 때문에 Engine과 Adapter의 경계가 흐릿해질 수 있다. 원칙은 유지하되, 과도하게 분리된 것 같으면 합쳐도 된다. 분리된 걸 합치는 게, 뒤섞인 걸 나중에 분리하는 것보다 쉽다.

### Factory 패턴

Motion.View를 하드코딩하는 대신 createMotionComponent 팩토리를 쓰는 이유:

- 새 컴포넌트 지원 시 umjik 내부 수정 불필요
- 사용자가 서드파티 컴포넌트도 감쌀 수 있음
- 모든 Motion 컴포넌트가 동일한 코드 경로를 타므로 버그가 한 곳에서만 발생

### AnimatePresence의 플랫폼 무관성

AnimatePresence의 핵심 문제: "React의 선언적 모델과 애니메이션의 시간적 모델 사이의 충돌"

이 충돌은 React를 쓰는 모든 플랫폼에서 동일하다. DOM이든 네이티브 뷰든, React가 컴포넌트를 unmount하면 렌더링이 멈추는 건 같다. 해결책도 동일하다: children을 state로 관리하면서 exiting 중인 컴포넌트를 React 트리에 유지한다.

### Reanimated 의존성

Legend Motion은 zero dependency를 위해 내장 Animated를 선택했다. umjik은 Reanimated를 선택한다.

Legend Motion의 판단: "대부분의 애니메이션은 opacity와 transform이면 충분하고, useNativeDriver로 성능이 괜찮다. 설치 편의성이 더 중요하다."

umjik의 판단: "framer-motion 호환을 위해 모든 속성을 지원해야 하고, useNativeDriver의 속성 제약은 호환성을 깨뜨린다. 타겟 사용자는 이미 Reanimated를 쓰고 있다."

두 판단 모두 타당하다. 다른 문제를 풀고 있을 뿐이다.

## 전제 6: TDD는 필수이다

모든 코드는 테스트로 검증합니다. 구현 전에 테스트를 먼저 작성합니다.

- **Engine 모듈은 순수 함수**이므로 단위 테스트로 완벽히 검증 가능하다. 외부 의존성이 없어 테스트 작성 비용이 낮고 신뢰도는 높다.
- **Adapter / Component는 mock 기반 통합 테스트**로 검증한다. `react-native`와 `react-native-reanimated`를 mock으로 대체하여 JS 환경에서 실행 가능하게 만든다.
- **Storybook으로 시각적 검증**을 수행한다. 자동화 테스트가 커버할 수 없는 실제 애니메이션 동작, 타이밍, 인터랙션을 눈으로 확인한다.

**검증 상태**: 구조적으로 안전. Engine 레이어가 순수 함수로 설계되어 있어 TDD 적용이 자연스럽다.
**리스크**: Adapter 테스트는 mock의 정확도에 의존한다. mock이 실제 Reanimated 동작과 다르면 테스트가 통과해도 런타임에서 실패할 수 있다. Storybook 시각적 검증이 이 gap을 보완한다.

## 구현 중 검증해야 할 것들

1. **framer-motion API와 RN의 불일치 지점 발견 시**, 무리하게 맞추지 말고 기록한다. 불일치가 사소하면 변환으로 해결하고, 근본적이면 API를 조정한다.
2. **Engine과 Adapter의 경계가 비실용적으로 느껴지면**, 합쳐도 된다. 단, "왜 합쳤는지"를 기록한다.
3. **추상화 오버헤드가 측정 가능한 수준이면**, 메모이제이션이나 구조 변경으로 해결한다.
4. **Reanimated 4의 CSS 애니메이션 문법이 Adapter를 단순화할 수 있는지** 검토한다.
